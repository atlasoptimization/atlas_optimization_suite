# Custom Domains

Custom domains are local extensions to the Data Science Deck. They do not modify canonical deck CSV files and they do not participate in handbook game-mode cycles by default.

## Actual Repository Layout

The current repository contains:

```text
custom_domains/
  generate_cards_all.sh
  generate_cards_readme
  scribus_generator/
    ScribusGenerator.py
    ScribusGeneratorBackend.py
    ...
  example_custom_domain/
    background/
    card_graphics/
    icons/
    output/
    placeholders/
    templates/
      card_data_example_domain.csv
      domain_metadata.txt
      export_pdf.py
      generate_cards_custom.sh
      template_example_domain.sla
```

## Editable vs Generated

User-editable:

- `custom_domains/<domain>/background/`
- `custom_domains/<domain>/card_graphics/`
- `custom_domains/<domain>/templates/card_data_*.csv`
- `custom_domains/<domain>/templates/domain_metadata.txt`
- `custom_domains/<domain>/templates/*.sla`, if you intentionally maintain Scribus templates

Treat as generator/pipeline files, not normal app data:

- `custom_domains/scribus_generator/`
- `custom_domains/generate_cards_all.sh`
- `custom_domains/<domain>/templates/generate_cards_custom.sh`
- `custom_domains/<domain>/templates/export_pdf.py`

Do not edit during normal app runtime:

- `custom_domains/<domain>/placeholders/`
- `custom_domains/<domain>/icons/`, unless deliberately changing the generator assets
- `custom_domains/<domain>/output/`, which is generated by the custom card pipeline

The web app does not write into `custom_domains/`. It only reads this folder during `pnpm sync:assets`.

## Metadata File

The example metadata file is:

```text
domain_name="example domain"
domain_folder_name="example_custom_domain"
domain_output_name="domain_output_name"
data_csv_name="card_data_example_domain.csv"
generate_script_name="generate_cards_custom.sh"
template_sla_name="template_example_domain.sla"
```

The sync parser accepts simple `key=value` or `key: value` lines, ignores blank lines and `#` comments, and strips surrounding quotes.

Required fields:

- `domain_name`
- `data_csv_name`

Recognized fields:

- `domain_name`
- `domain_folder_name`
- `domain_output_name`
- `data_csv_name`
- `generate_script_name`
- `template_sla_name`

## Card CSV

The custom-domain CSV lives under `templates/`, for example:

```text
custom_domains/example_custom_domain/templates/card_data_example_domain.csv
```

The example columns include:

- `nr`
- `cardname`
- `domain`
- `subdomain`
- `summary`
- `twin`
- `keywords`
- `effect_good`
- `effect_bad`
- `effect_mod`
- `question`
- `story`
- `art`
- `ring`
- `uniform_bg`
- `zonal_bg`
- `thematic_bg`
- `symbol`
- `author`
- `homepage`

Rows are mapped into the same `DeckCard` model used by canonical cards. Custom-domain cards receive:

- `origin = "custom-domain"`
- `customDomainId = <domain folder slug>`
- `id = custom:<domainSlug>:<nr-cardSlug>`

## Generated PDFs and Previews

Generated PDFs are read from:

```text
custom_domains/<domain>/output/
```

The sync step matches PDFs using the card number and card name, for example:

```text
ML_Card_Example Dom_1_Domain Master.pdf
```

During `pnpm sync:assets`, matched PDFs are copied into:

```text
apps/web/public/deck/custom-domains/<domainSlug>/pdfs/
```

PDF previews are generated into:

```text
apps/web/public/deck/custom-domains/<domainSlug>/pdf-previews/
```

Page 1 is used as the front preview and page 2 as the back preview. If no PDF exists, the card still works and renders in text-card mode.

## App Consumption

`pnpm sync:assets` now:

1. Builds the canonical manifest as before.
2. Scans `CUSTOM_DOMAINS_ROOT`, defaulting to `./custom_domains`.
3. Ignores generator folders such as `scribus_generator`.
4. Reads each domain’s `templates/domain_metadata.txt`.
5. Reads the configured `templates/card_data_*.csv`.
6. Copies matched art/PDF assets into public deck custom-domain folders.
7. Generates PDF previews when possible.
8. Appends discovered custom-domain card metadata to `apps/web/public/deck/manifest.json`.

Sync only discovers content packs. It does not activate them in the app.

Runtime behavior:

- Discovered custom domains are inactive by default.
- Enable or disable them in the app from the left card browser: `Custom > Custom Domains > Manage custom domains`.
- Active custom domains appear in browser/search and receive draw/discard piles after the canonical six piles.
- Inactive custom domains are hidden from normal browser/search/pile controls.
- Placed cards from an inactive custom domain still render because their metadata remains in the manifest.
- Right-click pile actions reuse the existing pile inspector/draw/discard logic.
- Canonical mode cycles remain Source, Structure, Chameleon, Void, Volition, Aspect.
- Folder-based custom domains are distinct from in-app custom cards. In-app custom cards remain under `Custom Cards`.

## Workflow

1. Copy `custom_domains/example_custom_domain/`.
2. Rename the copied folder.
3. Edit `templates/domain_metadata.txt`.
4. Edit `templates/card_data_*.csv`.
5. Add images under `card_graphics/` and backgrounds under `background/`.
6. Run the custom-domain generator to create PDFs in `output/`.
7. Run `pnpm sync:assets`.
8. Start the app.
9. Open `Custom > Custom Domains > Manage custom domains` in the left card browser and enable the content pack.

If a generator script change seems necessary, document it before changing it. The app integration treats the generator as an external production pipeline.
